# 事件信号监控引擎

> 当前版本：`v0.14.0` · 统一 60 秒循环 · 模拟交易 · 无需 Docker

这是一个面向加密市场的事件聚合、量化信号分析和模拟交易系统。它把行情、新闻、热榜、交易所公告、预测市场和数学模型统一为同一条决策链，并保存产生信号时的依据、模拟交易结果与交易后复盘数据。

项目默认只运行 `paper-alert-only` 模式，不包含真实下单逻辑。

## 当前运行架构

```text
内置 RSS / NewsNow / GDELT / 交易所公告 / Polymarket / 可选 Whale Alert
                              │
                              ▼
                统一事件、去重、来源分级、时效分析
                              │
                              ▼
             行情模型 + 消息方向 + 成本 + 自适应开仓门槛
                              │
                              ▼
                 模拟账户、仓位管理、交易链路快照
                              │
                              ▼
             每 X 笔交易复盘、影子验证、权重晋升或回滚
```

完整监控每 60 秒执行一次。若某一轮超过 60 秒，系统等待该轮完成后再进入下一轮，不会并发叠加请求或同时写入账户状态。

## 主要能力

| 模块 | 当前实现 |
| --- | --- |
| 消息聚合 | 内置 12 个 RSS、7 个 NewsNow 热榜、GDELT、交易所公告和可选 Whale Alert |
| 行情数据 | 价格、资金费率、未平仓量和波动状态 |
| 预测市场 | 按市场 ID 持续追踪 Polymarket 双方概率、比例、成交量和流动性，直到市场关闭 |
| 消息质量 | 关键词过滤、来源分级、标题近似聚类、跨源佐证、时间戳和半衰期衰减 |
| 公告过滤 | 过滤例行上币、新现货交易对和新永续合约；保留下架、暂停、安全与风险公告 |
| 数学模型 | GBM、GARCH、HMM、泊松事件到达、贝叶斯后验校准和 Markowitz 配置 |
| 开仓门槛 | 以含手续费和滑点的保本胜率为起点，按校准误差、样本量、行情状态、波动和方向分歧动态调整 |
| 模拟账户 | 支持现货与合约，计入手续费、滑点和资金费率，展示净值、回撤、滚动夏普和逐笔盈亏 |
| 交易复盘 | 保存开仓因子、权重、模型版本、相关消息与决策快照；按每 X 笔已平仓交易触发批量复盘 |
| 权重迭代 | 候选权重先做按时间顺序切分的影子验证，达到样本和改善门槛后才允许晋升，并支持回滚 |

消息来源固定在源码中。仪表盘只保留“启用状态”和“保留关键词”配置，避免用户误把外部服务地址当作项目必需配置。

## 不依赖哪些外部进程

项目直接请求和解析内置 RSS，并直接调用 NewsNow 缓存接口，因此不要求额外启动：

- Docker
- RSSHub
- DailyHotApi
- TrendRadar
- NewsNow 服务进程

World Monitor、TrendRadar 等项目提供的聚合思路可以作为设计参考，但本项目运行时不依赖它们。

Whale Alert 仅作为可选兼容来源。没有 API Key 时不会产生系统告警，也不会阻断其他来源和数学模型。

## 本地运行

### 环境要求

- Windows 10/11
- Node.js 20 或更高版本
- PowerShell 5.1 或更高版本

### 安装

```powershell
npm ci
Copy-Item .env.example .env
```

`.env` 中的 Whale Alert API Key 可以留空。

### 启动统一监控循环

```powershell
npm run signal:loop
```

### 启动仪表盘

```powershell
npm run dashboard
```

浏览器访问：`http://127.0.0.1:8788/`

停止本地监控循环：

```powershell
npm run signal:stop
```

## 服务器持续运行

当前服务器部署把两个职责分开管理：

- `event-signal-monitor.timer`：每 60 秒触发一次完整监控。
- `event-signal-dashboard.service`：持续提供仅监听服务器本机地址的仪表盘。

服务器循环不依赖本地电脑、浏览器或 VPN。本地电脑关机、关闭浏览器或者停止 SSH 隧道，都不会停止服务器上的监控和模拟账户。

常用只读检查命令：

```bash
systemctl status event-signal-monitor.timer
systemctl status event-signal-dashboard.service
curl http://127.0.0.1:8788/api/status
```

仪表盘中的“循环运行”状态会优先读取 PID；Linux systemd 定时器没有常驻循环 PID 时，则根据最新报告的时间判断。报告超过设定的陈旧阈值后才会显示停止，避免把正常的 systemd 定时任务误报为未运行。

注意：服务器服务在线和模拟账户已启动是两个不同状态。模拟账户仍需在仪表盘中启动一次；启动后，账户状态由服务器保存并由后续监控轮次持续更新。

## 消息处理规则

1. 所有来源先转换为统一事件结构。
2. 按标题相似度合并为故事簇，避免同一新闻被多次放大。
3. 重复报道优先保留来源等级更高的代表项；同等级优先保留本轮响应更快的来源。
4. 其余来源只作为交叉佐证，不重复增加事件数量。
5. 每条消息同时保存事件发生时间和采集时间，并按来源类型使用不同半衰期计算时效权重。
6. 热榜保存首次出现时间、当前排名、上一轮排名、最佳排名、出现次数和排名动量。

Polymarket 市场进入监控后，会持续记录隐含概率、比例和流动性变化，直到上游明确标记关闭。历史观测采用分层留存：最近 3 小时分钟级、最近 7 天小时级、最长 5 年日级。

## 交易后复盘边界

交易复盘衡量的是因子与结果之间的预测关联，不是因果证明。

- 复盘使用扣除手续费、滑点和资金费率后的净 R。
- 展示因子关联、MFE、MAE、持仓路径和候选权重。
- 功能加入前建立的旧持仓没有开仓时因子快照，平仓后会标为不可归因样本。
- 系统不会使用事后数据伪造开仓时的决策链路。
- 自动晋升默认受到最小样本量、验证改善幅度和单次权重变化上限约束。

## 数据与安全

运行报告、账户状态、聚合器配置、预测市场历史、日志和本地凭据写入：

```text
.runtime/event-signal-monitor/
```

`.runtime/` 和 `.env` 均被 Git 忽略。真实 API Key、SSH 私钥、服务器密码和账户凭据不得写入源码、README、Issue 或提交记录。

项目只绑定本机仪表盘地址。若未来加入真实交易，必须另行实现权限隔离、请求签名、资金限额、审计日志、密钥轮换和人工授权，不能直接复用模拟账户接口作为实盘接口。

## 已知限制

- 外部新闻和行情接口可能限流、延迟或临时不可用。
- 消息来源缺失时系统会继续使用数学模型，但这不等于数据完整。
- 预测市场赔率反映参与者定价，不是真实概率证明。
- 因子权重优化可能受到样本不足、市场状态变化和过拟合影响。
- 历史模拟结果不能证明未来收益。

无证据表明新闻聚合、大模型推理、Polymarket 赔率或当前数学模型能够稳定盈利。
